# v0.8.0 Hardening Design: Typed Errors, Timeout, Cancellation

> **Status**: Draft
> **Created**: 2026-03-07
> **Context**: mcp-protocol-sdk v0.7.0 ships bidirectional Server-to-Client requests.
> This design addresses robustness gaps identified by cross-referencing the official
> Rust SDK (`rmcp`) and TypeScript SDK (`@modelcontextprotocol/sdk`).

## Goal

Make mcp-protocol-sdk "production-grade" by closing the gap between the current
OCaml implementation and the reference SDKs. Focus on correctness and resilience,
not feature breadth.

## Architecture Overview

Three independent hardening layers, each deliverable independently:

```
Layer 1: Typed Error Hierarchy     (lib/mcp_error.ml)
Layer 2: Request Timeout + Cancel  (eio/server.ml, eio/client.ml)
Layer 3: Transport Robustness      (eio/stdio_transport.ml, lib/jsonrpc.ml)
```

---

## Gap Analysis (v0.7.0 vs Reference SDKs)

| Gap | OCaml v0.7.0 | Rust SDK | TypeScript SDK | Priority |
|-----|-------------|----------|----------------|----------|
| Typed error hierarchy | `(result, string) result` everywhere | `RmcpError > ServiceError > ClientInitializeError` | `SdkError` (local) + `ProtocolError` (wire) | P0 |
| Request timeout | error_codes defines -32002, never used | `Timeout { timeout: Duration }` variant | `DEFAULT_REQUEST_TIMEOUT_MSEC = 60_000` | P0 |
| Cancellation | `notifications/cancelled` constant exists, no runtime | `Cancelled { reason }` in ServiceError, `CancellationToken` | AbortSignal + `notifications/cancelled` handler | P1 |
| Progress subscription | Server sends, client ignores | `ProgressDispatcher` with token subscribe/unsubscribe | `onprogress` callback per request, `resetTimeoutOnProgress` | P2 |
| `message_of_yojson` catch-all | `with _ ->` swallows all exceptions | N/A (serde returns typed Result) | N/A (Zod schema validation) | P0 |
| Transport max_size | Hardcoded 1MB | Configurable | N/A (stream-based) | P2 |
| Write error handling | `Eio.Flow.copy_string` can raise, uncaught in transport | Transport error variants | N/A | P1 |
| Initialization resilience | Simple request/response | Handles notifications/pings during handshake loop | Protocol base handles interleaved messages | P1 |
| Capability enforcement | None | Capability-gated dispatch | `enforceStrictCapabilities` option | P2 |

## Layer 1: Typed Error Hierarchy

### Problem

Every function returns `(result, string) result`. Callers cannot pattern-match
on error kinds — they can only log the string message. This makes error recovery
impossible (e.g., distinguishing timeout from method-not-found from transport closed).

### Design

Two error types mirroring the TS/Rust split:

```ocaml
(** lib/mcp_error.ml *)

(** Protocol errors: cross the wire as JSON-RPC error responses. *)
type protocol_error = {
  code: int;            (** JSON-RPC error code from Error_codes *)
  message: string;
  data: Yojson.Safe.t option;
}

(** SDK errors: local errors that never cross the wire. *)
type sdk_error =
  | Not_connected
  | Connection_closed of string   (** reason *)
  | Request_timeout of float      (** seconds waited *)
  | Cancelled of string option    (** reason *)
  | Send_failed of string         (** transport error detail *)
  | Parse_error of string         (** JSON/JSON-RPC parse failure *)
  | Internal of string            (** catch-all for unexpected *)

(** Combined error type. *)
type t =
  | Protocol of protocol_error
  | Sdk of sdk_error

val to_string : t -> string
val protocol : code:int -> message:string -> ?data:Yojson.Safe.t -> unit -> t
val timeout : float -> t
val cancelled : ?reason:string -> unit -> t
val connection_closed : string -> t
```

### Migration Strategy

1. New functions use `(result, t) result` where `t = Mcp_error.t`
2. Add `Mcp_error.to_string_result : ('a, t) result -> ('a, string) result` bridge
3. Existing public API keeps `string` error for backward compatibility via thin wrappers
4. Internal functions migrate to typed errors incrementally

### Reference

- **Rust SDK**: `ServiceError` with `McpError`, `TransportSend`, `TransportClosed`, `Cancelled`, `Timeout` variants
- **TypeScript SDK**: `SdkError` class with `SdkErrorCode` enum (`REQUEST_TIMEOUT`, `CONNECTION_CLOSED`, `SEND_FAILED`, etc.)

## Layer 2: Request Timeout and Cancellation

### 2A: Request Timeout

#### Problem

`Client.read_response` and `Server.server_read_response` loop forever waiting
for a matching response ID. If the peer hangs or is slow, the caller blocks
indefinitely.

#### Design

```ocaml
(** Default timeout: 60 seconds (matches TypeScript SDK) *)
val default_timeout : float  (* 60.0 *)

(** Per-request timeout option *)
type request_options = {
  timeout: float option;    (** seconds; None = use default *)
}
```

**Implementation**: Use `Eio.Time.with_timeout` (or `Eio.Time.Timeout.run`)
around the read_response loop. On timeout:
1. Return `Error (Mcp_error.Sdk (Request_timeout elapsed))`
2. Send `notifications/cancelled` to peer with the timed-out request ID

```ocaml
(* In Client.send_request *)
let send_request t ~method_ ?params ?(timeout=default_timeout) () =
  let id = ... in
  match Stdio_transport.write t.transport msg with
  | Error e -> Error (Mcp_error.Sdk (Send_failed e))
  | Ok () ->
    let clock = Eio.Stdenv.clock t.env in  (* needs env reference *)
    match Eio.Time.with_timeout clock timeout (fun () -> read_response t id) with
    | Ok result -> result
    | Error `Timeout ->
      (* Best effort: notify peer we gave up *)
      ignore (send_notification t
        ~method_:Notifications.cancelled
        ~params:(Some (`Assoc [
          ("requestId", id_to_yojson id);
          ("reason", `String "Request timed out")
        ])) ());
      Error (Mcp_error.Sdk (Request_timeout timeout))
```

**Trade-off**: This requires passing `Eio.Stdenv.clock` to Client/Server.
Alternative: accept a `timeout_fn` callback at creation. Chosen approach:
pass `env` or at least `clock` at construction time, similar to how `stdin`/`stdout`
are passed today.

### 2B: Cancellation (notifications/cancelled)

#### Problem

The spec defines `notifications/cancelled` for aborting in-flight requests.
Neither client nor server handles this notification.

#### Design — Client Side

When client receives `notifications/cancelled` from server while waiting:
- Match the `requestId` from the notification params
- Abort the in-flight read_response loop
- Return `Error (Mcp_error.Sdk (Cancelled reason))`

#### Design — Server Side

When server receives `notifications/cancelled` from client:
- The notification arrives during dispatch (while a handler is running)
- OCaml's Eio supports structured concurrency — cancellation requires
  the handler to be running in a fiber that can be cancelled
- **v0.8.0 scope**: Record the cancellation, but handler runs to completion
  (handlers are synchronous in the current design)
- **Future**: Fiber-based handlers with `Eio.Cancel.check` support

#### Notification Format (MCP Spec)

```json
{
  "jsonrpc": "2.0",
  "method": "notifications/cancelled",
  "params": {
    "requestId": 42,
    "reason": "User requested cancellation"
  }
}
```

### Reference

- **Rust SDK**: `CancellationToken` from `tokio_util`, `serve_with_ct()` variant
- **TypeScript SDK**: `AbortSignal` per request, `_oncancel` handler in Protocol base class

## Layer 3: Transport Robustness

### 3A: Fix `message_of_yojson` Exception Swallowing

#### Problem

```ocaml
(* lib/jsonrpc.ml line 134 *)
  with _ ->
    Error "Failed to parse JSON-RPC message"
```

This catches ALL exceptions including `Out_of_memory`, `Stack_overflow`,
and any programming bugs. It masks real errors.

#### Fix

```ocaml
  with
  | Yojson.Json_error msg -> Error (Printf.sprintf "JSON parse error: %s" msg)
  | exn -> Error (Printf.sprintf "Unexpected parse error: %s" (Printexc.to_string exn))
```

Catch only expected exceptions. For truly unexpected ones, include the
exception name in the error message instead of silently discarding it.

### 3B: Transport Write Error Handling

#### Problem

`Eio.Flow.copy_string` in `Stdio_transport.write` can raise `Eio.Io` exceptions
(e.g., broken pipe). The current code only checks `t.closed` but doesn't catch
write failures.

#### Fix

```ocaml
let write t msg =
  if t.closed then Error "Transport is closed"
  else begin
    try
      let json = Mcp_protocol.Jsonrpc.message_to_yojson msg in
      let line = Yojson.Safe.to_string json ^ "\n" in
      Eio.Flow.copy_string line t.sink;
      Ok ()
    with
    | Eio.Io _ as ex ->
      t.closed <- true;
      Error (Printf.sprintf "Transport write error: %s" (Printexc.to_string ex))
  end
```

### 3C: Configurable max_size

#### Current

```ocaml
let reader = Eio.Buf_read.of_flow stdin ~max_size:(1024 * 1024)
```

#### Fix

```ocaml
let default_max_size = 1024 * 1024  (* 1MB *)

let create ~stdin ~stdout ?(max_size=default_max_size) () =
  let reader = Eio.Buf_read.of_flow stdin ~max_size in
  { reader; sink = ...; closed = false }
```

### 3D: Initialization Handshake Resilience

#### Problem

`Client.initialize` sends initialize request and expects the next response
to be the matching reply. If the server sends a notification or ping first
(allowed by spec), the client may fail or loop unexpectedly.

#### Current Behavior

`read_response` already handles this — it dispatches server requests and
skips notifications while waiting. But it doesn't handle the case where
a `notifications/cancelled` arrives during initialization.

#### Fix

The `read_response` loop is already mostly correct for initialization.
Add explicit handling for `notifications/cancelled` in the loop.

### 3E: Read Error Recovery

#### Problem

`Stdio_transport.read` catches `End_of_file` but not `Eio.Io` exceptions.
A broken pipe on stdin would crash the server/client.

#### Fix

```ocaml
let rec read t =
  if t.closed then None
  else
    match Eio.Buf_read.line t.reader with
    | line -> (* ... existing logic ... *)
    | exception End_of_file -> None
    | exception (Eio.Io _ as ex) ->
      t.closed <- true;
      Some (Error (Printf.sprintf "Transport read error: %s" (Printexc.to_string ex)))
```

## Constraints and Trade-offs

### Clock Dependency

Adding timeout requires `Eio.Time.clock`. Two options:

**A) Pass `clock` at construction** (chosen):
```ocaml
val create : stdin:_ Eio.Flow.source -> stdout:_ Eio.Flow.sink ->
  clock:_ Eio.Time.clock -> t
```
- Pro: Explicit, testable (mock clock), no global state
- Con: Breaking change to `Client.create` and `Server.run` signatures

**B) Pass `env`**: More flexible but couples to `Eio.Stdenv`
- Rejected: Too broad, harder to test

### Backward Compatibility

- `Mcp_error.t` is new; existing `(result, string) result` API stays
- `Client.create`/`Server.run` gain optional `?clock` parameter
  - Without clock: no timeout (v0.7.0 behavior)
  - With clock: default 60s timeout
- `Stdio_transport.create` gains optional `?max_size` parameter

### What's NOT in v0.8.0

- HTTP/SSE transport (separate package, separate timeline)
- Resource subscribe/unsubscribe (types exist, server dispatch deferred)
- Fiber-based handler cancellation (requires async handler redesign)
- Capability enforcement (nice-to-have, not a correctness issue)
- Progress subscription on client side (types exist, runtime deferred)

## Risks

1. **Clock parameter breaking change**: `Client.create` and `Server.run` signatures change.
   Mitigation: Make `?clock` optional with `None` = no timeout.

2. **Timeout in tests**: Mock clock needed for deterministic timeout tests.
   Eio provides `Eio_mock.Clock` for this.

3. **Transport error recovery**: After a write error, transport is marked closed.
   No reconnection. This matches Rust/TS behavior for stdio transports.
